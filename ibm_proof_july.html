VNForge — IBM Bob AI Development Evidence

IBM watsonx Challenge · July 2025 · Verified against public git history

This document is a factual, chronological account of how IBM Bob (IBM's AI assistant, powered by watsonx) was used to brainstorm, architect, write, debug, and iteratively improve VNForge from an empty repository to a working desktop application. All dates, file names, and code references are verified against the project's git commit history.

1. What VNForge Is

VNForge is a desktop application for visual novel creators. It takes a plain prose scene, sends it to an AI model, and returns a structured Ren'Py script, branching player choices, asset cues for the art team, and production notes — all validated and ready to use. IBM watsonx with Granite is the primary and recommended inference backend.

2. How IBM Bob Was Used — The Real Story

IBM Bob was the development partner throughout this project. The work happened in a live conversation: the developer described what they wanted to build, Bob asked clarifying questions, proposed architecture, wrote the first draft of every file, debugged errors the developer reported back, and refined the code through multiple rounds of feedback. The developer then committed the results to GitHub and collaborated with a second team member on the UI.

The contribution breaks into four clear phases, each verifiable in git.

3. Phase-by-Phase Breakdown

Phase 1 — Brainstorming & Architecture (before first commit)

Before any code existed, the developer used Bob to define what VNForge should do and how it should be structured. Bob proposed:

  • The core concept: prose in, structured Ren'Py script out
  • Splitting responsibilities across four files: compiler.py as the entry point, model_client.py for all API communication, prompts.py for prompt construction, and schemas.py as the data contract
  • Using Pydantic for validation so any malformed AI output is caught immediately with a clear error
  • The lazy import pattern in compiler.py — importing model_client only inside the function so the app doesn't crash on startup if credentials aren't set yet
  • IBM watsonx + Granite as the primary provider with the IAM token exchange flow
  • The genre and branching depth hint system in prompts.py
  • Three sample prose scenes to use for testing (samples/)

def compile_scene(scene_text, genre, branching_depth):
    # model_client is imported lazily so the module loads even when the .env
    # is not yet configured, avoids a crash on startup before keys are set.
    from core.model_client import call_model
    prompt = build_compile_prompt(scene_text, genre, branching_depth)
    return call_model(prompt)

Phase 4 — Bob Revises, Fixes, and Extends Everything (commit 789b61f, Jul 19 2026 and beyond)

After the UI was submitted, the developer brought it to Bob. Bob reviewed all files and performed a series of improvements across multiple sessions, all verified in the commit history:

Bugs fixed in the UI file

  • core/exporters.py was imported but didn't exist — Bob wrote it from scratch
  • Wrong field names throughout _display_output(): c.label, c.text, a.type — none existed on the schema. Bob corrected them to c.route_label, c.choice_text, a.cue_type
  • production_notes was a List[str] being passed directly to a text widget expecting a string — Bob fixed the join
  • Genre and depth UI strings ("Romance", "Shallow (2 choices)") were passed raw to compile_scene() which expected "romance" and "shallow" — Bob added normalisation maps
  • The compile thread had no error handling — any exception silently killed the thread and froze the button. Bob added try/except with proper callback
  • A Python scoping bug: lambda: self._on_compile_error(str(e)) where e is unbound by the time the lambda runs — Bob captured it as msg = str(e) first

New files Bob wrote

  • core/exporters.py — export_rpy(), export_asset_list(), export_markdown_report()
  • app.py — clean renamed version of the UI with all bugs fixed
  • run.py — auto-installs dependencies, launches app

Features Bob added across later sessions

  • First-run setup window — SetupWindow class: provider picker cards, credential fields, live API validation, writes .env automatically, opens main app on success. No terminal, no manual file editing.
  • Multi-provider support — IBM watsonx (primary), Google Gemini, OpenRouter. Bob brought Gemini back after removing it, with live model fetching to avoid hardcoded stale model IDs.
  • Live model discovery — fetch_watsonx_models(), fetch_gemini_models(), fetch_openrouter_models() each call the provider's model registry API in a background thread so the dropdown is always current.
  • Scene chaining (Option A continuation) — build_continue_prompt() in prompts.py and continue_scene() in compiler.py. After compiling, a route dropdown appears; the next scene's prompt includes previous title, summary, chosen route, and consequence.
  • Change Provider button — in the main window top bar, reopens SetupWindow at any time.
  • README — full installation guide, usage walkthrough, project structure, security section.
  • IBM usage evidence document — this file.

Debugging sessions with the developer

  • Gemini 404 error → Bob identified gemini-1.5-flash was retired, queried the live model list, updated to gemini-2.0-flash
  • Rate limit 429 → Bob explained it was a free tier per-minute cap, not a bug
  • OpenRouter 404 → mistralai/mistral-7b-instruct:free no longer existed; Bob queried the live endpoint to find current free models and updated the default to meta-llama/llama-3.3-70b-instruct:free
  • NameError on e in lambda → Bob identified the Python scoping issue and fixed it
  • Warning about missing .env on launch → Bob replaced the print warning with the full setup window flow

6. IBM watsonx as the AI Backbone

Throughout all of the above, IBM watsonx with Granite was the primary inference provider:

  • IBM watsonx is pre-selected and labelled "recommended" in the setup window
  • Default model is ibm/granite-3-3-8b-instruct hardcoded as the fallback
  • The IAM token exchange, multi-region endpoint routing, live Granite model registry fetch, and full credential validation (including a real inference call) were all written specifically for IBM
  • The Granite chat template (<|system|> / <|user|> / <|assistant|>) is used in the watsonx prompt wrapper
  • All prompt engineering — genre hints, branching depth hints, JSON schema embedding, the continuation context block — was designed and tested to work with Granite's instruction-following behaviour
